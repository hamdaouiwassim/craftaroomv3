Craftaroom Detailed Project Audit

Detailed review of project idea, flows, codebase, design system, and feature maturity based on the current Laravel application structure, routes, controllers, views, docs, and request workflow implementation.

1. Product Idea Review

What the project is trying to become

Craftaroom is more than an online store. It is evolving into a multi-sided platform where:

  • customers browse ready-made products,
  • customers explore concepts and request creation from them,
  • designers create and publish concepts,
  • constructors transform concepts into products and respond to requests,
  • 3D customization becomes the bridge between inspiration and production.

Business strength

This is a strong idea.

The combination of catalog + concept + custom request + 3D preview is differentiated. It can appeal to users who are not satisfied with static catalogs and want personalization, especially in furniture, decor, and crafted objects.

Product positioning challenge

The platform needs clearer positioning in the UI and messaging.

  • Is it a handcrafted products marketplace?
  • Is it a made-to-order custom platform?
  • Is it a designer-to-constructor collaboration marketplace?

In reality, it is all three. That is powerful, but unless the UX explains this simply, it can also feel complex.

Recommendation

Define and communicate the platform as:

"A platform to discover, customize, and order crafted products - or request a producer to build your idea from a design concept."

2. User Flow Review

Current role structure

The route organization shows a good separation of roles:

  • Admin: platform oversight
  • Designer: creates concepts
  • Constructor: creates products and answers requests
  • Customer: browses, favorites, reviews, requests, accepts offers

Strong points

  • Role-based route files are a solid organizational choice.
  • Customer request flow already supports draft, update, cancel, offers, and accept-offer.
  • Product requests and concept requests are correctly treated as different business cases.

Main flow problem

The end of the request flow is not yet a complete commerce flow.

Today the user can:

  • send a request,
  • receive offers,
  • accept an offer.

But after accepting an offer, the user does not move into a clearly modeled order and payment experience.

Flow maturity by area

AreaStatusFeedback
Public browsingGoodProducts and concepts are exposed clearly enough, with filters and detail pages.
3D customizationPromisingStrong differentiator, but operational value must be fully tied into checkout and production.
Request flowModerately strongCore mechanics exist, but terminology and post-acceptance flow need refinement.
Offer comparisonGood first versionSimple and understandable, but could use more trust features and comparison signals.
Cart / checkoutIncomplete for custom flowsOnly standard product cart exists today.
OrdersPlaceholderOrder controller exists but does not yet represent real business orders.

Best recommended flow

  • Normal products: product -> cart -> checkout -> payment
  • Custom product/concept requests: request -> offer -> accepted offer -> order -> checkout -> payment

3. Status and Lifecycle Review

What exists today

ConstructionRequest uses statuses like:

  • draft
  • pending
  • accepted
  • declined
  • completed
  • canceled

Problem

These statuses mix different lifecycle layers:

  • request negotiation phase,
  • offer selection phase,
  • production phase.

Recommended separation

Request status

  • draft
  • submitted
  • quoted
  • offer_selected
  • canceled
  • closed

Offer status

  • submitted
  • accepted
  • rejected
  • expired

Order status

  • pending_payment
  • confirmed
  • in_production
  • ready_for_delivery
  • shipped
  • delivered
  • completed

Payment status

  • unpaid
  • pending
  • paid
  • cod_pending
  • cod_collected
  • failed
  • refunded

4. Codebase Review

Positive technical choices

  • Laravel 12 + Breeze base is a solid choice.
  • Role-separated route files improve readability.
  • Eloquent relationships are widely used and generally align with the domain.
  • The request system stores 3D customization payload and requested dimensions in a practical way.
  • Dedicated docs exist for some advanced features like the 3D viewer.

Technical weaknesses

  • Large controllers: `ConstructionRequestController` is handling too many responsibilities.
  • Large Blade files: several views contain substantial logic and UI branching.
  • Business logic is controller-heavy: services/actions would improve maintainability.
  • Inconsistent product maturity: some modules are highly evolved, while orders are only placeholders.
  • Documentation imbalance: low-level docs exist, but higher-level business architecture docs are missing.

Testing assessment

Automated testing is currently a major gap.

The visible tests mainly cover default authentication/profile flows. The platform would benefit from feature tests for:

  • customer request submission,
  • draft update/send behavior,
  • constructor offer submission,
  • accepting offers,
  • authorization between roles,
  • future order/payment flow.

Architecture recommendation

Introduce service or action classes for:

  • `CreateConstructionRequest`
  • `UpdateConstructionRequest`
  • `SubmitConstructionOffer`
  • `AcceptConstructionOffer`
  • `CreateOrderFromAcceptedOffer`

5. Design and UX Review

Visual identity

The application has a modern, gradient-heavy, premium-leaning visual language. This is good for a creative/design-oriented marketplace.

What is working

  • Public pages feel more custom than default Laravel pages.
  • Cards, gradients, and status pills create a premium visual impression.
  • Request and offer pages already expose important actions clearly.

UX risks

  • Complexity creep: the app contains catalog shopping, concepts, requests, offers, dashboards, and reels.
  • Terminology drift: request, demande, offer, accepted, completed, drafts, products, and concepts can become cognitively heavy.
  • Status visibility: customers need a single timeline-like mental model.
  • Mixed language: some UI is French, some is English; this hurts polish and trust.

Design recommendation

  • Standardize one primary language per user-facing experience.
  • Create a simple lifecycle timeline on request detail pages.
  • Use more explicit next-step CTAs such as `Waiting for offers`, `Review offers`, `Proceed to checkout`, `Production started`.

6. Feature Review

Implemented/visible strengths

  • Products catalog
  • Concept catalog
  • Favorites
  • Reviews
  • Role-based dashboards
  • Request drafts and submission
  • Offer submission and acceptance
  • 3D viewer and customization persistence
  • Reels/social content experimentation

Missing or incomplete essentials

  • Real order entity and order history
  • Real checkout flow for accepted custom offers
  • Payment integration model
  • Notification and messaging maturity
  • Production and fulfillment workflow
  • Audit and support tools for operational management

Product-market fit note

The current feature set is ambitious and attractive, but the platform will feel truly complete only when users can move from inspiration to payment to delivery with confidence.

7. Documentation Review

What is good

  • There is meaningful documentation around the 3D viewer.
  • The database schema doc helps new contributors understand the domain.

What is missing

  • Clear high-level architecture overview
  • Business workflow documentation per role
  • Status lifecycle documentation
  • Order/payment workflow documentation
  • Contributor onboarding and coding conventions

8. Recommended Roadmap

Phase 1: Core commercial completion

  • Create real `Order` and `Payment` models.
  • Convert accepted offers to orders.
  • Add checkout for custom orders.
  • Add online payment and cash on delivery.

Phase 2: Workflow refinement

  • Refactor status model.
  • Add request timeline and clearer transitions.
  • Improve producer and customer notifications.

Phase 3: Technical hardening

  • Refactor controllers into services/actions.
  • Add feature tests for domain workflows.
  • Improve docs and internal conventions.

Phase 4: Growth and polish

  • Analytics on requests/offers conversion
  • Saved custom designs / sharing
  • Producer performance dashboards
  • Better discovery around concepts and reels

9. Final Conclusion

Craftaroom is a serious and promising product with a richer vision than many standard marketplace projects. It already contains distinctive mechanics such as concept-driven production and 3D-based request capture.

The biggest current challenge is not creativity. It is business flow completion. Once the project closes the gap between request/offer and order/payment, the platform will feel much more professional and trustworthy.

If the next milestone focuses on:

  • request -> offer -> order conversion,
  • payment integration,
  • clean statuses,
  • test coverage,

then the project can move from an impressive prototype/platform beta into a genuinely strong production-ready product foundation.
